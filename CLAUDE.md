# Claude Code instructions — Workout Tracker

## Project at a glance

A personal, single-device, two-profile workout tracker. Web/PWA, no auth, no backend. Local IndexedDB + JSON export/import for durability. Hosted on GitHub Pages.

Strong Curves is one of two **built-in routine templates** that ship with the app — it is **not** the app's identity. The app is a generic workout tracker that can run any routine.

Owner: Joshua Stafford-Haworth (JSHPhysics).

**Always read `SCOPE.md` before any major design decision.** This file is the operating manual; SCOPE.md is the spec.

## Conventions

### Files & naming
- Components in PascalCase, hooks in camelCase prefixed `use`, files named after their default export.
- Domain types live in `src/types/`. Dexie schemas reference these types.
- Seed data lives in `src/seed/`. Strong Curves seed is generated by `scripts/build-seed.py` from the source `.xlsx`; never hand-edit the generated TS.
- Storage layer in `src/db/` — single `db.ts` for the Dexie instance, plus per-entity query modules.
- Domain logic that's pure (PR detection, plate calculation, e1RM, streak) lives in `src/domain/` as plain TS modules with unit tests.

### State
- Dexie is the source of truth for all domain data. Query via `useLiveQuery` from `dexie-react-hooks`.
- Zustand is **only** for ephemeral UI state — active timer, the set being entered right now, modal visibility, the in-flight rest-timer state. Never put domain data in Zustand.
- No prop-drilling state that belongs in Dexie.

### TypeScript
- Strict. No `any`, no `@ts-ignore`, no `as` casts except at well-documented boundaries (e.g. parsing imported JSON).
- Discriminated unions for measurement types, set types, block types, PR types.
- Exhaustiveness checks via `assertNever` helper.

### Styling
- Tailwind classes only. No separate CSS files except global resets and PWA splash.
- Dark mode default; light mode follows `prefers-color-scheme`.
- Honour `prefers-reduced-motion` (especially for the PR celebration modal).

### Inputs
- Custom number steppers for weight/reps/time/RPE. **Do not use native `<input type="number">`** — fiddly on mobile.
- Tap targets ≥ 48 px.

## UX principles

- One primary action per screen.
- Inline editing > modal flows wherever possible.
- Never block the workout flow on a sync, save, or write — all writes are optimistic and async.
- The session screen is the most important surface. Optimise for it ruthlessly.
- Today screen offers three paths in priority order: continue active routine → pick a routine → free session.

## Pure-domain modules

These should be plain TS, no React, no Dexie, fully unit-tested:

- **`domain/e1rm.ts`** — Epley estimated 1RM
- **`domain/plate-calculator.ts`** — given target weight, bar weight, and inventory, return per-side plate loadout + closest achievable
- **`domain/pr-detection.ts`** — given an exercise's full set log history, return what PRs the latest set(s) achieved
- **`domain/streak.ts`** — given session timestamps and a timezone, return current and longest streak
- **`domain/volume.ts`** — set-log → volume calculations, including muscle-group apportionment with the secondary-muscles weighting
- **`domain/set-classification.ts`** — auto-warmup heuristic logic

Keep these honest and tested — UI bugs are cheap, domain bugs in PR/plate/volume calculations destroy trust in the data.

## Data integrity rules

- **Never delete a profile** without an explicit JSON backup confirmation step.
- **Schema migrations are additive** — add new versions via Dexie's `version().stores()` chain. Never edit a past version.
- **Destructive actions** require confirmation + a 5-second undo toast where feasible.
- **Session timestamps are immutable.** `startedAt` set on Start, `completedAt` on Finish. Editing historical set logs is permitted; editing timestamps is not.
- **Built-in routines are read-only.** Editing prompts to fork.
- **Backup file format is versioned** (`schemaVersion`). Bump on any breaking change to export shape and write a migration in the importer.
- **PRs are derived state** — always recoverable by replay over set logs. On JSON import, PRs are recomputed from set logs (do not trust imported `PRRecord` rows blindly).

## Backup & durability

The user's data lives in IndexedDB, which **can** be cleared. Durability comes from the JSON file. Build the nag UX seriously:

- Header banner if last backup > 7 days
- Modal block on entering Settings if > 30 days
- Post-session prompt: *"Back up now?"* with stale info
- File System Access API for silent auto-writes where supported (desktop Chrome/Edge)
- Triggered download fallback everywhere else

Track `lastBackupAt` per profile.

## Build & deploy

- `pnpm dev` — local dev
- `pnpm build` — produces fully static `dist/`

## Out of scope (do not build in v1)

- Social, sharing, leaderboards
- Body measurements, progress photos

- Cardio metrics (heart rate, distance)
- Auto-progression engine
- Mood / energy / sleep tracking
- Photo / video exercise demos
- Per-exercise barbell override

If asked to add any of the above, push back with a reference to SCOPE.md §11 and propose deferring.
